# 2026-07 Systematic Code Review

## Objective

Establish repeatable validation for the customization layer, then review and remediate risks in patch generation, release automation, persisted state, automated account actions, and frontend contracts.

## Upstream Baseline

The pinned PR baseline lives in `compatibility/upstream.env`. Pull requests and pushes to `main` use those tags for deterministic checks. Scheduled and manually selected latest-mode runs resolve the newest upstream releases to detect drift.

## Severity

| Level | Definition |
| --- | --- |
| P0 | Credential exposure, unauthorized destructive action, unrecoverable data loss, or compromised release |
| P1 | Incorrect automated account action, persistent state corruption, concurrency failure, or broken release path |
| P2 | User-visible correctness, compatibility, performance, or maintainability risk with a practical workaround |
| P3 | Low-impact hardening, documentation, or code-quality issue |

## Completion Criteria

- Required PR checks cover repository, Core, and Management validation.
- Release workflows cannot publish before the same Core and Management validation succeeds.
- Scheduled latest-upstream checks report patch drift independently of the pinned PR baseline.
- All P0 and P1 findings are fixed or explicitly accepted with an owner and rationale.
- Each behavioral fix includes a regression test and a fresh-upstream validation result.
